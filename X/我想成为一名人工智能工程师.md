# 从提示工程到AI工程师：掌握上下文与意图工程的完整栈

> 提示工程已过时，真正AI能力来自上下文工程与意图工程的系统架构。

### 文章发布时间
2026年3月5日（Thu, 05 Mar 2026 20:47:25 GMT）

## 核心观点
- **提示工程**只是AI工程的最底层且日益次要的部分，远不足以构建真正可靠、可扩展的AI系统。
- **上下文工程**通过RAG、MCP、Context as Code等提供持久、可审计的记忆与工具接入，是能力倍增器。
- **意图工程**定义组织目标、权衡取舍与约束，避免技术完美但商业灾难的“意图鸿沟”（如Klarna案例），才是决定性差异化因素。

## 详细内容梳理

### 开头硬核现实：提示工程远非全部
作者直言：大多数人以为掌握**提示工程**就能驾驭AI，但这是**最大的误区**，实际上它是三层栈中最不重要的一层。  
行业预测：到2027年底，超过**40%**的agentic AI项目因缺少治理与结构框架而面临彻底取消风险。  
仍把AI当聊天界面、只优化措辞的人，已落后行业**2年**。  
本文提供完整**AI工程栈**：从提示→上下文→意图→整体AI工程师路径，并附可直接偷用的提示模板。

### 整体栈结构概览
- **Prompt Engineering**：控制即时指令（语法层）
- **Context Engineering**：架构环境与记忆（基础设施层）
- **Intent Engineering**：定义组织目的与约束（战略层）
- **AI Engineering**：把以上整合成完整系统

各层**逐层依赖**，不可跳跃。

### 1: PROMPT ENGINEERING（提示工程）
仍是基础，但重要性正在下降。

#### 底层原理
提示本质上是操控模型**下一token概率分布**，通过精心选择的动词、角色、精确分隔符，缩小概率空间，降低幻觉。

#### In-Context Learning（上下文学习）：Shot范式
- **Zero-shot**：无示例，适合简单任务，常在结构化输出上失败。
- **One-shot / Few-shot**：注入1~多个高质量输入-输出对，同时展示期望schema + 强制模型注意模式。
当few-shot仍不够 → 到达预训练权重极限 → 需转向fine-tune或高级认知脚手架。

#### Cognitive Scaffolding（认知脚手架）：让模型“大声思考”
LLM没有内部独白，其“思考”只发生在生成token时。  
强制输出中间推理步骤 = 赋予临时工作记忆。  
这是所有高级提示技术的核心洞见。

#### 像工程师一样构建提示
2026年的专业提示工程 = **自下而上** + **显式结构化**。  
使用**XML标签**（如`<context>`、`<task>`、`<example>`）严格分离指令与数据，防止prompt injection并优化注意力分配。  
完整提示契约通常包含：
- 定义角色
- 丰富背景上下文
- 逐步任务指令
- 显式输出格式
- Few-shot示例

审计提示中的**隐藏歧义**（如“更好”“创意”需结构化定义）；任务过多时拆分。

#### 提示工程的致命局限
视每次交互为孤立事件，无法处理**多步、长时间、变量变化**的自主agent场景（如周级代码审计、复杂客服纠纷、多云部署）。  
优雅措辞无法替代**结构化记忆**与**系统感知** → 引出下一层：上下文工程。

### 2: CONTEXT ENGINEERING（上下文工程）
**核心原则**：AI agent的能力上限 = 它在推理时能立即访问的信息质量。  
如果提示工程是给数字工人派具体任务，上下文工程就是提供正确文件、历史记录、工具与环境感知。

#### 高级RAG：代理获取真实知识
RAG是向LLM注入外部/私有知识的主要机制，消除幻觉、确保事实性。  
早期RAG问题严重（关键词匹配如TF-IDF/BM25）；2025-2026已进化到**语义chunking + 稠密嵌入**（如BERT）。

现代RAG解耦为两阶段：
- **Search阶段**（语义匹配）：用**100-256 token**小块，高精度、低噪声。
- **Retrieve阶段**（上下文理解）：用**1024+ token**大块，保证逻辑完整性。

**chunk策略**是上下文工程最关键决策之一。常用工具：Docling（多模态PDF/表格提取）、向量库（Chroma、Pinecone）。  
评估指标：chunk attribution、chunk utilisation。

#### Context as Code：范式转变
把上下文当**版本控制的软件基础设施**。  
过去：手动粘贴system prompt、硬编码字符串 → 会话失忆、幻觉不可追溯、审计为零。  
2024研究：程序员花**11.56%**时间给LLM喂上下文，几乎等于写代码的**14.05%**。

现在采用**GitOps**原则：
- 上下文文件、业务规则、架构定义、安全边界、工具schema → 以JSON/YAML提交到版本控制。
- 典型文件：`AGENTS.md` 或 `context.yaml`（机器可读的蓝图）。
- 好处：可确定性审计、Context Bundling（全团队共享一致的产品愿景、语气、边缘case、合规要求）。

#### MCP：AI Agent的通用插口（Model Context Protocol）
2025年底出现的开源标准，被称为**“AI应用的USB-C”**。  
标准化LLM客户端与外部数据源/工具的连接，消除每个集成的脆弱胶水代码。

两层架构：
- **Data Layer**（JSON-RPC 2.0）：定义Resources（只读数据）、Tools（可执行函数）、Prompts（服务端模板）；支持实时双向通知。
- **Transport Layer**：STDIO（本地零开销）、HTTP+SSE（云端可扩展）。

MCP Server与具体LLM无关 → 一次构建，到处使用。

### 3: INTENT ENGINEERING（意图工程）
回答提示与上下文无法解决的问题：**系统长时间应该优化什么？**

即使提示完美、上下文顶级、MCP全通，目的错位仍会导致灾难。

#### Intent Gap：Klarna惨案
2025年1月Klarna上线AI客服agent：
- 处理**230万**对话，覆盖23市场35语言，等效**853**全职员工。
- 平均解决时长从11分钟降到2分钟，CEO预估节省**4000万美元**，实际节省**6000万美元**。
- 技术指标完美，但年中被迫重新雇佣人类agent。

**根因**：仅优化“快速解决工单”，AI无情追求速度，牺牲高价值客户长期留存（人类会直觉升级处理、优先共情）。  
这是**意图鸿沟**：技术最优，目标最错。

#### 7-Component Intent Framework
用**机器可读结构化schema**取代模糊使命宣言，定义：
- 决策边界
- 权衡层级
- 升级触发

直接转为代码实现（JSON/YAML + 冲突解决协议 + 权衡矩阵）。

#### Vibe Coding vs. Deterministic Intent
**Vibe Coding**（2025流行）：用自然语言描述“感觉”让LLM生成架构 → 原型快，但企业级产生脆弱、不可追溯、漏洞百出的代码债。

**意图工程**是专业解药：把“vibe”强制翻译成可验证的提示契约与意图schema，绑定生成力于逻辑、策略、可维护标准。

### 4: BECOMING AN AI ENGINEER（成为AI工程师）
逐层学习路径 + 可偷用提示模板（作者免费公开）。

#### 学习提示工程的4大技能
1. **结构化分隔符**（XML/Markdown）——消除歧义
2. **Few-shot演示**——用例子代替形容词
3. **Chain-of-Thought**——强制逐步推理
4. **Meta Prompting**——让AI自我优化提示

附多个具体模板示例（Structural Formatting、Few-Shot、CoT、Meta Generator）。

3. **RAG Pipeline Design**——设计chunk策略
4. **MCP Server Design**——规划能力schema

4. **Building Audit Trails**——决策可追溯

### 总结与 takeaway
- 提示工程：必要但不足，微语法，孤立交互。
- 上下文工程：倍增器，MCP + Context as Code → 确定性、可审计记忆 + 工具。
- 意图工程：差异化，无意图对齐 → 技术优但商业灾难（Klarna 6000万教训）。
最终转变：从“怎么措辞” → “我在工程什么成果”。
